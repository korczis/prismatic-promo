Release

infrastructure

OTP release packaging that bundles compiled application, ERTS runtime, and configuration into a self-contained deployable unit.

Definition

An OTP Release is a self-contained, deployable package that bundles a compiled Elixir/Erlang application together with the Erlang Runtime System (ERTS), all compiled dependencies, boot scripts, and configuration into a single directory that can run on a target machine without requiring Elixir, Erlang, or any development tools to be installed. Releases are the standard production deployment artifact for BEAM applications, analogous to Docker images for containerized services or JARs for Java applications, but with the unique addition of including the entire runtime system.

The mix release command (introduced in Elixir 1.9) handles the release assembly process: compiling all applications and their dependencies to BEAM bytecode, copying the ERTS from the build machine, generating boot scripts that define the startup order based on the application dependency graph, and creating wrapper scripts for start, stop, remote console, and health check operations. The resulting release directory is hermetic -- it contains everything needed to run, making it ideal for deployment to bare-metal servers, virtual machines, or Docker containers.

Releases support runtime configuration through config/runtime.exs, which is evaluated at boot time (not compile time), enabling the same release artifact to be deployed to different environments by varying environment variables. This separation of build artifact from environment configuration is a key principle of the twelve-factor app methodology and is essential for CI/CD pipelines that build once and deploy to staging and production sequentially.

Implementation in Prismatic Platform

The Prismatic Platform builds releases for deployment to Fly.io via multi-stage Docker containers. The umbrella application produces a single release named prismatic containing all 90 apps with their supervision trees. Runtime configuration loads environment variables for database URLs, API keys, secret key bases, cluster settings, and feature flags through config/runtime.exs. The CI/CD pipeline in GitLab builds the release artifact, runs the full test suite against it, and deploys to staging before production.

The release includes custom boot scripts in the rel/ directory for health checks, database migrations, and seed data loading. The rel/overlays directory contains runtime files that are copied into the release directory, including the Fly.io process manifest and custom vm.args for BEAM VM tuning (scheduler count, process limit, atom table size).

Release Assembly Process

The release build follows a deterministic pipeline:

Source Code (90 apps)
      |
      v
  mix compile --warnings-as-errors
      |  All .ex files compiled to .beam bytecode
      v
  mix release prismatic
      |  1. Resolve application dependency graph
      |  2. Copy compiled .beam files
      |  3. Copy ERTS from build machine
      |  4. Generate boot scripts (.script, .boot)
      |  5. Copy config/runtime.exs
      |  6. Apply rel/overlays
      |  7. Generate start/stop scripts
      v
  _build/prod/rel/prismatic/
      |-- bin/prismatic       # Start/stop/remote_console scripts
      |-- lib/                # Compiled application .beam files
      |-- erts-*/             # Erlang Runtime System
      |-- releases/
      |     |-- 0.1.0/
      |     |     |-- start.boot    # Boot script
      |     |     |-- sys.config    # Compile-time config
      |     |     |-- runtime.exs   # Runtime config
      |     |     |-- vm.args       # BEAM VM arguments
      |-- tmp/                # Runtime temporary files

Release Configuration

# mix.exs release configuration
def project do
  [
    releases: [
      prismatic: [
        include_executables_for: [:unix],
        applications: [runtime_tools: :permanent],
        steps: [:assemble, :tar],
        cookie: "prismatic_cookie_#{Mix.env()}"
      ]
    ]
  ]
end

Docker Multi-Stage Build

The release is built inside a multi-stage Docker build for minimal image size:

# Stage 1: Build
FROM hexpm/elixir:1.19.0-erlang-27.0-debian-bookworm AS builder

WORKDIR /app
ENV MIX_ENV=prod

COPY mix.exs mix.lock ./
COPY config config
COPY apps apps

RUN mix deps.get --only prod && \
    mix deps.compile && \
    mix compile --warnings-as-errors && \
    mix release prismatic

# Stage 2: Runtime (minimal image)
FROM debian:bookworm-slim AS runner

RUN apt-get update -y && \
    apt-get install -y libstdc++6 openssl libncurses5 locales && \
    apt-get clean && rm -rf /var/lib/apt/lists/*

WORKDIR /app
COPY --from=builder /app/_build/prod/rel/prismatic ./

ENV PHX_HOST=prismatic-prod.fly.dev
ENV PORT=4000

CMD ["bin/prismatic", "start"]

The release binary supports several operational commands:

bin/prismatic daemonStart as background daemonTraditional deployment
bin/prismatic stopStop running instanceGraceful shutdown
bin/prismatic remoteConnect IEx to running nodeLive debugging

bin/prismatic evalExecute one-off commandDatabase migrations
bin/prismatic pidGet OS PID of running instanceProcess management
# Run database migrations at deploy time
bin/prismatic eval "PrismaticStorage.Release.migrate()"

# Remote shell into production for debugging
bin/prismatic remote

# Health check via RPC
bin/prismatic rpc "PrismaticWeb.HealthCheck.status()"

Hot Code Reload

While hot code reload is a BEAM capability, modern Elixir releases favor immutable deployments (build new release, deploy, restart) over in-place upgrades:

ApproachMethodRiskUse Case
Immutable DeployBuild new release, replace oldLow (rollback = deploy previous)Standard production
Hot Upgrademix release --upgrade with appupsMedium (state migration required)Zero-downtime critical
Rolling RestartDeploy to nodes sequentiallyLow (nodes restart independently)Clustered deployments on Fly.io

The Prismatic Platform uses rolling restarts on Fly.io -- new release images are deployed to instances one at a time, maintaining availability through the load balancer while each instance restarts with the new code.
