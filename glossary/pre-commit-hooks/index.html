Pre-Commit Hooks

architecture

Automated quality enforcement executed before each git commit

Definition

Pre-Commit Hooks are git hook scripts that execute automatically before each commit is finalized. They run a multi-phase quality pipeline including compilation verification, Credo static analysis, test execution for modified files, Quality Debt Point scanning, risk pattern detection, and predictive regression prevention. If any phase fails, the commit is blocked until the issue is resolved. The hooks are the platform's first line of defense against quality regression.

Hook Phases

The pre-commit pipeline executes phases sequentially. A failure in any phase halts execution and blocks the commit:

PhaseNameCheckBlocking
1Compilationmix compile --warnings-as-errorsYes
2Credo Analysismix credo --strict on modified filesYes
3Test Executionmix test for modified modules and their testsYes
4QDP ScanQuality Debt Point detection in staged filesYes
5Risk Pattern DetectionAnti-patterns (length() > 0, Process.sleep, missing @spec)Yes
6Unsafe Access CheckUnsafe map access (map.field vs Map.get/2)Yes
7High-Risk File GuardExtra scrutiny for critical files (configs, migrations)Yes
8Promo Template Validation<div> wrapper check for section list templatesConditional
9Predictive RegressionHistorical pattern matching against known regression signaturesYes

Risk Patterns Detected

The hooks scan for specific code patterns known to cause issues:

BLOCKED PATTERNS:
  - length(list) > 0     --> Use Enum.any?/1 or pattern match
  - Process.sleep(...)    --> Use receive/after or :timer
  - map.field             --> Use Map.get(map, :field) or Map.fetch!
  - Missing @spec         --> All public functions require typespecs
  - TODO / FIXME          --> Must be resolved before commit
  - IO.inspect(...)       --> Debug output not allowed in commits

MONITORED PATTERNS:
  - Changes to mix.exs    --> Dependency changes flagged for review
  - Changes to migrations --> Schema changes flagged for review
  - Changes to configs    --> Configuration drift detection

 --no-verify?  --> L4 SUPREME REVIEW (FORBIDDEN)

The --no-verify flag is absolutely forbidden by Session Discipline protocol. Its use triggers L4 Supreme Review escalation -- the highest violation level before platform lockdown. This is not a guideline; it is a hard enforcement rule with zero exceptions.

Context

Within the Prismatic Platform, pre-commit hooks are defined in .githooks/ and include the Predictive Pre-Commit quality protection system. The hooks are installed automatically and cannot be bypassed through normal workflow. They detect risk patterns before code enters the repository, serving as the local complement to GitLab CI/CD remote validation. The current platform state of 0 Quality Debt Points and 100/100 quality score is maintained directly by this enforcement layer. Every commit that enters the repository has passed all nine phases.
