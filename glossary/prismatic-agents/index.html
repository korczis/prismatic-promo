Prismatic Agents

agents

Agent runtime engine managing 434 AIAD agents across 14 domains

Definition

Prismatic Agents (prismatic_agents) is the runtime engine application within the Prismatic Platform's umbrella that manages the complete lifecycle, coordination, dispatch, and monitoring of 434 AIAD-compliant agents across 14 operational domains. The application provides agent registration, discovery by capability or domain, message routing, telemetry emission, and graceful degradation through OTP-supervised processes. Each agent is defined by an AIAD specification file that declares its tier level (L1-L5), operational domain, capability set, authority scope, and behavioral constraints, creating a uniform runtime contract that the engine enforces.

Prismatic Agents represents the convergence of AI agent orchestration with Erlang/OTP's battle-tested process management model. Rather than treating agents as stateless functions invoked on demand, the platform models each agent as a supervised process with its own state, mailbox, and lifecycle -- enabling persistent context, autonomous behavior, inter-agent communication, and fault-tolerant operation that survives individual agent failures without cascading to the broader system.

Overview

The agent runtime engine addresses several challenges that emerge at scale in multi-agent AI systems. Discovery and dispatch must be fast -- O(1) -- to avoid becoming a bottleneck when hundreds of agents process concurrent requests. State management must be isolated so that a misbehaving agent cannot corrupt another's context. Coordination must be asynchronous and message-based to avoid deadlocks in complex multi-agent workflows. Monitoring must be comprehensive to detect degraded agents before they produce unreliable output. And the entire system must degrade gracefully when individual agents fail, restart, or are taken offline for updates.

The 434 agents span 14 operational domains: security (Color-Team agents), intelligence (OSINT agents), quality (guardian and gate agents), evolution (autoevolve and autoheal agents), tactical (ARCHER, Supreme Commander, Delta Force), infrastructure (supervisor, storage, API agents), content (promo enhancer, documentation agents), analysis (epistemic pipeline, NABLA agents), compliance (NIS2, ZKB, OWASP agents), development (code generation, refactoring agents), testing (regression, property-based, integration agents), monitoring (telemetry, health, performance agents), deployment (release, CI/CD, Docker agents), and research (GARDEN, knowledge extraction agents).

Technical Details

Agent Registry Architecture

The Agent Registry provides O(1) lookup of agent definitions using ETS (Erlang Term Storage) as the backing store:

defmodule PrismaticAgents.Registry do
  @moduledoc """
  ETS-backed agent registry providing O(1) lookup by name,
  domain, tier, and capability. Registry is populated at
  boot time from AIAD agent definition files.
  """

  use GenServer

  @table_name :agent_registry
  @index_table :agent_index

  def start_link(opts) do
    GenServer.start_link(__MODULE__, opts, name: __MODULE__)
  end

  @impl true
  def init(_opts) do
    :ets.new(@table_name, [:named_table, :set, :public, read_concurrency: true])
    :ets.new(@index_table, [:named_table, :bag, :public, read_concurrency: true])

    agent_count = load_agent_definitions()
    {:ok, %{agent_count: agent_count, loaded_at: DateTime.utc_now()}}
  end

  @spec lookup(String.t()) :: {:ok, agent_definition()} | {:error, :not_found}
  def lookup(agent_name) do
    case :ets.lookup(@table_name, agent_name) do
      [{^agent_name, definition}] -> {:ok, definition}
      [] -> {:error, :not_found}
    end
  end

  @spec by_domain(atom()) :: [agent_definition()]
  def by_domain(domain) do
    :ets.lookup(@index_table, {:domain, domain})
    |> Enum.map(fn {_key, agent_name} ->
      {:ok, def} = lookup(agent_name)
      def
    end)
  end

  @spec by_tier(1..5) :: [agent_definition()]
  def by_tier(tier) do
    :ets.lookup(@index_table, {:tier, tier})
    |> Enum.map(fn {_key, agent_name} ->
      {:ok, def} = lookup(agent_name)
      def
    end)
  end

  @spec by_capability(atom()) :: [agent_definition()]
  def by_capability(capability) do
    :ets.lookup(@index_table, {:capability, capability})
    |> Enum.map(fn {_key, agent_name} ->
      {:ok, def} = lookup(agent_name)
      def
    end)
  end

  defp load_agent_definitions do
    Path.wildcard(".aiad/agents/*.agent.md")
    |> Enum.map(&parse_agent_definition/1)
    |> Enum.each(fn definition ->
      :ets.insert(@table_name, {definition.name, definition})
      :ets.insert(@index_table, {{:domain, definition.domain}, definition.name})
      :ets.insert(@index_table, {{:tier, definition.tier}, definition.name})

      Enum.each(definition.capabilities, fn cap ->
        :ets.insert(@index_table, {{:capability, cap}, definition.name})
      end)
    end)
    |> length()
  end
end

Agent Lifecycle

defmodule PrismaticAgents.AgentProcess do
  @moduledoc """
  Generic agent process implementing the AIAD agent lifecycle.
  Each agent runs as a supervised GenServer with its own state,
  message queue, and telemetry emission.
  """

  use GenServer, restart: :transient

  require Logger

  @type state :: %{
    name: String.t(),
    tier: 1..5,
    domain: atom(),
    status: :initializing | :active | :degraded | :suspended | :terminated,
    context: map(),
    message_count: non_neg_integer(),
    error_count: non_neg_integer(),
    started_at: DateTime.t()
  }

  def start_link(definition) do
    GenServer.start_link(__MODULE__, definition, name: via_tuple(definition.name))
  end

  @impl true
  def init(definition) do
    emit_telemetry(:agent_started, definition)

    {:ok,
     %{
       name: definition.name,
       tier: definition.tier,
       domain: definition.domain,
       status: :active,
       context: %{},
       message_count: 0,
       error_count: 0,
       started_at: DateTime.utc_now()
     }}
  end

  @impl true
  def handle_call({:dispatch, message}, _from, state) do
    case execute_agent_logic(state, message) do
      {:ok, result} ->
        new_state = %{state | message_count: state.message_count + 1}
        emit_telemetry(:message_processed, state)
        {:reply, {:ok, result}, new_state}

      {:error, reason} ->
        new_state = %{state | error_count: state.error_count + 1}
        emit_telemetry(:message_error, state, %{reason: reason})
        {:reply, {:error, reason}, maybe_degrade(new_state)}
    end
  end

  defp maybe_degrade(%{error_count: count} = state) when count > 10 do
    Logger.warning("Agent #{state.name} degraded: #{count} errors")
    emit_telemetry(:agent_degraded, state)
    %{state | status: :degraded}
  end

  defp maybe_degrade(state), do: state

  defp via_tuple(name), do: {:via, Registry, {PrismaticAgents.ProcessRegistry, name}}

  defp emit_telemetry(event, state, metadata \\ %{}) do
    :telemetry.execute(
      [:prismatic_agents, event],
      %{timestamp: System.monotonic_time()},
      Map.merge(%{agent: state.name, domain: state.domain, tier: state.tier}, metadata)
    )
  end
end

Implementation in Prismatic Platform

Within the Prismatic Platform's umbrella, prismatic_agents serves as the central agent coordination hub. The application depends on prismatic_storage_core for trait definitions and prismatic_storage_ets for registry backing. It integrates with the broader platform through several key interfaces:

Agent Dispatch: The dispatcher resolves incoming requests to the appropriate agent based on capability matching, tier authority, and domain routing. Multi-agent workflows are orchestrated through message passing between agent processes, with the dispatcher handling fan-out and aggregation patterns.

Telemetry Integration: Every agent lifecycle event (start, message processed, error, degradation, restart) emits telemetry events under the [:prismatic_agents, *] namespace. The Quality Floor Guardian consumes these events to detect quality regressions in agent behavior.

AIAD Compliance: All 434 agents follow the AIAD (AI Agent Integration and Definition) standard, which specifies the format for agent definition files, capability declarations, authority scopes, and behavioral constraints. The registry validates AIAD compliance at load time.

Domain Distribution: The 14 domains provide organizational structure and isolation boundaries. Agents within a domain can communicate freely, while cross-domain communication follows explicit routing rules through the dispatcher.

Prismatic Agents' primary differentiator is its foundation on OTP, which provides battle-tested process supervision, fault isolation, and hot code reloading that purpose-built agent frameworks cannot match. The tradeoff is the requirement that agents run within the BEAM ecosystem.

Best Practices

Domain Isolation: Group agents by operational domain and enforce isolation boundaries. A security agent should not directly access quality agent state -- all cross-domain communication should flow through the dispatcher.

Tier Authority: Respect the tier hierarchy (L1-L5) in agent interactions. Lower-tier agents should not override decisions made by higher-tier agents. The tier system prevents authority conflicts in multi-agent workflows.

Graceful Degradation: Design agents to degrade gracefully rather than crash hard. An agent experiencing transient errors should enter degraded status with reduced capabilities rather than crashing and restarting with lost context.

Telemetry Coverage: Emit telemetry events for all significant agent actions, not just errors. Comprehensive telemetry enables proactive monitoring and trend analysis by the Quality Floor Guardian.

Stateless Where Possible: Minimize per-agent state. Agents that rely heavily on accumulated state are harder to restart and more vulnerable to state corruption. Persist critical state to storage backends and reload on restart.

Use Cases

Security Operations: Color-Team agents (Red, Blue, Purple, White, Gray, Black) conduct adversarial simulation, defensive monitoring, synthesis, verification, boundary exploration, and theoretical threat modeling through coordinated multi-agent workflows.

Quality Management: Guardian agents monitor quality metrics, gate agents enforce quality thresholds, and evolution agents drive autonomous quality improvement through coordinated campaigns.

Intelligence Gathering: OSINT agents conduct open-source intelligence operations, aggregating data from multiple sources with the NABLA Infinity framework's signal plurality requirement.

Platform Evolution: AutoEvolve and AutoHeal agents drive autonomous platform improvement, identifying quality regressions and applying fixes without human intervention.

Tactical Operations: ARCHER, Supreme Commander, and specialist agents orchestrate complex multi-step operations requiring coordination across multiple domains and authority levels.
