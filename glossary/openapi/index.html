OpenAPI

api

Specification standard for describing REST APIs in machine-readable format, enabling automatic documentation generation, client SDK creation, and API validation.

Definition

OpenAPI (formerly known as the Swagger Specification) is an industry-standard, language-agnostic specification for describing RESTful APIs in a machine-readable format using JSON or YAML. Maintained by the OpenAPI Initiative under the Linux Foundation, the specification defines a structured document format that captures every aspect of an API's surface area---available endpoints, HTTP methods, request parameters, request and response body schemas, authentication mechanisms, error responses, and operational metadata. This single-source-of-truth approach enables an entire ecosystem of tooling to operate automatically: documentation generators produce interactive API explorers, code generators create type-safe client SDKs in dozens of languages, and validation middleware enforces request/response conformance at runtime.

The specification has evolved through three major versions. OpenAPI 2.0 (Swagger) established the core concepts, OpenAPI 3.0 introduced components (reusable schemas), improved security definitions, and support for multiple content types, and OpenAPI 3.1 aligned the schema format with JSON Schema 2020-12 for full compatibility with the broader JSON Schema ecosystem. The 3.0 specification is currently the most widely adopted version and the one used by the Prismatic Platform's API layer.

An OpenAPI specification serves as both a design artifact (created during API planning to define the contract before implementation) and a runtime artifact (generated from code annotations to document what has been implemented). The design-first approach ensures API consumers and producers agree on the contract before development begins, while the code-first approach eliminates documentation drift by generating the specification directly from source code.

Specification Structure

An OpenAPI 3.0 document is organized into several top-level sections, each describing a different aspect of the API:

SectionPurposeRequired
openapiSpecification version (e.g., "3.0.3")Yes
infoAPI title, version, description, contactYes
serversBase URLs for API environmentsNo
pathsEndpoint definitions with operationsYes
componentsReusable schemas, parameters, responsesNo
securityAuthentication requirementsNo
tagsLogical grouping of operationsNo
# Simplified OpenAPI 3.0 specification
openapi: "3.0.3"
info:
  title: Prismatic Platform API
  version: "1.0.0"
  description: Auto-introspecting REST API for Prismatic Platform

servers:
  - url: http://localhost:4004/api/v1
    description: Development

paths:
  /perimeter/discover:
    post:
      summary: Discover attack surface for a domain
      operationId: perimeterDiscover
      tags: [perimeter]
      requestBody:
        required: true
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/DiscoverRequest'
      responses:
        '200':
          description: Discovery results
          content:
            application/json:
              schema:
                $ref: '#/components/schemas/AttackSurface'

components:
  schemas:
    DiscoverRequest:
      type: object
      required: [domain]
      properties:
        domain:
          type: string
          example: "example.com"
    AttackSurface:
      type: object
      properties:
        assets:
          type: array
          items:
            $ref: '#/components/schemas/Asset'

Schema and Type System

OpenAPI's type system, based on JSON Schema, provides a rich vocabulary for describing data structures with validation constraints:

TypeFormatElixir EquivalentExample
string(default)String.t()"hello"

integerint32integer()42

Advanced schema features include:

  • oneOf/anyOf/allOf: Polymorphic types and schema composition
  • discriminator: Automatic type selection based on a field value
  • nullable: Fields that accept null values
  • enum: Restricted value sets
  • pattern: Regex validation for strings
  • minimum/maximum: Numeric range constraints

OpenApiSpex Integration

The Prismatic Platform uses OpenApiSpex, an Elixir library that generates OpenAPI 3.0 specifications from Phoenix controller annotations and Elixir type specifications:

defmodule PrismaticApi.Schemas.SecurityRating do
  require OpenApiSpex
  alias OpenApiSpex.Schema

  OpenApiSpex.schema(%{
    title: "SecurityRating",
    description: "Security rating for an asset",
    type: :object,
    required: [:grade, :score, :asset_id],
    properties: %{
      grade: %Schema{type: :string, enum: ["A", "B", "C", "D", "E", "F"]},
      score: %Schema{type: :integer, minimum: 300, maximum: 900},
      asset_id: %Schema{type: :string, format: :uuid},
      factors: %Schema{type: :object, additionalProperties: true},
      measured_at: %Schema{type: :string, format: :"date-time"}
    }
  })
end

OpenApiSpex provides three key capabilities for the platform:

  1. Schema Definition: Elixir modules define request/response schemas using OpenApiSpex macros
  2. Request Validation: Incoming requests are automatically validated against the schema before reaching controller logic
  3. Spec Generation: The complete OpenAPI 3.0 specification is generated at runtime and served as JSON

Context in Prismatic

The prismatic_api application implements a unique auto-introspecting API architecture that combines OpenAPI with Elixir's runtime introspection capabilities. Rather than manually defining API endpoints, the system automatically discovers all public functions across Prismatic* facade modules and exposes them as documented REST API endpoints.

Auto-Discovery Pipeline: At boot time, a Scanner module uses Code.fetch_docs/1, Code.Typespec.fetch_specs/1, and Module.__info__/1 to enumerate all public functions, extract their type specifications, and build an endpoint registry cached in ETS.

Type Mapping: Elixir @spec type annotations are automatically translated to JSON Schema types for the OpenAPI specification. For example, @spec discover(String.t()) :: {:ok, map()} | {:error, term()} generates a schema with a string parameter and an object response.

Generic Dispatch: A single Plug-based controller resolves {app, action} path parameters to module.function(args) calls, eliminating the need for per-endpoint controller modules.

Endpoints:

RouteMethodDescription
/api/v1/healthGETHealth check
/api/v1/endpointsGETList all discovered endpoints
/api/v1/:app/:actionGET/POSTGeneric dispatch (GET for 0-2 params, POST otherwise)
/api/openapiGETOpenAPI 3.0 JSON specification
/api/swaggeruiGETInteractive Swagger UI documentation
Scanner --> Registry (ETS) --> DispatchController --> safe_apply(Module, :function, args) --> JSON
               |
         TypeMapper --> OpenApiSpex Schema
               |
         ApiSpec --> SwaggerUI

This architecture means that adding a new public function to any Prismatic facade module automatically creates a new API endpoint with full OpenAPI documentation, request validation, and Swagger UI entry---zero manual configuration required.

API Versioning

OpenAPI supports API versioning through multiple strategies, each with distinct trade-offs:

StrategyImplementationProsCons
URL Path/api/v1/resourceClear, cacheableURL pollution
HeaderAccept: application/vnd.api.v2+jsonClean URLsHidden versioning
Query Parameter/api/resource?version=2SimpleNot RESTful

The Prismatic Platform uses URL path versioning (/api/v1/) for clarity and cacheability. The OpenAPI specification includes version metadata in the info section, and the oasdiff tool detects breaking changes between specification versions during CI/CD.

Related Terms

  • API Gateway - Entry point that serves and enforces the OpenAPI specification

  • GraphQL - Alternative API paradigm with its own introspection mechanism
  • Typespec - Elixir type annotations that feed the auto-generated OpenAPI spec
