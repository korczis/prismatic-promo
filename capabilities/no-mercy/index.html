NO MERCY

DOCTRINE

Zero tolerance for incomplete implementations, quality violations, and untested code

Doctrine Overview

NO MERCY is the enforcement arm of the Prismatic platform doctrine. It establishes absolute, non-negotiable quality standards that apply to every line of code, every agent operation, and every platform decision.

NO MERCY: Complete execution or no delivery. Zero tolerance for compromise.

The Four Pillars

1. Zero Incomplete Implementations

Principle: Every feature, fix, or change must be 100% complete before delivery.

ForbiddenRequired
Partial implementationsFull feature completion
"Phase 1" with missing partsComplete MVP or nothing
"Will fix later"Fix now or don't ship
Feature flags hiding incomplete codeWorking code only

Enforcement:

# Pre-commit check
def validate_completeness(changeset) do
  cond do
    has_todo_comments?(changeset) -> {:error, :incomplete}
    has_stub_functions?(changeset) -> {:error, :incomplete}
    has_placeholder_values?(changeset) -> {:error, :incomplete}
    missing_tests?(changeset) -> {:error, :incomplete}
    true -> :ok
  end
end

2. Zero Stubs, Mocks, or TODOs

Principle: Production code contains only production-ready implementations.

Violation TypeStatus
# TODO: implement laterBLOCKED
# FIXME: known bugBLOCKED
def placeholder_function, do: :not_implementedBLOCKED
raise "Not implemented"BLOCKED
Mock objects in production codeBLOCKED

Detection:

# Automated scanning in pre-commit
grep -r "TODO\|FIXME\|NotImplemented\|:not_implemented" apps/
# Exit code 1 if any matches = commit blocked

3. 100% Test Coverage Mandatory

Principle: Every line of code must have corresponding test coverage.

Coverage TypeTargetEnforcement
Line coverage100%HARD
Branch coverage100%HARD
Function coverage100%HARD
Regression testsEvery bug fixHARD

Test Requirements:

# Every public function needs:
# 1. Happy path test
# 2. Error case test
# 3. Edge case test (boundaries, nil, empty)
# 4. Property-based test (where applicable)

defmodule PaymentProcessorTest do
  use ExUnit.Case
  use StreamData

  # Happy path
  test "processes valid payment" do
    assert {:ok, _} = PaymentProcessor.process(%{amount: 100})
  end

  # Error case
  test "rejects negative amount" do
    assert {:error, :invalid_amount} = PaymentProcessor.process(%{amount: -1})
  end

  # Edge case
  test "handles zero amount" do
    assert {:error, :zero_amount} = PaymentProcessor.process(%{amount: 0})
  end

  # Property test
  property "never loses money" do
    check all amount <- positive_integer() do
      {:ok, tx} = PaymentProcessor.process(%{amount: amount})
      assert tx.amount == amount
    end
  end
end

4. Zero Warnings in Production

Principle: The codebase compiles and runs with zero warnings.

Warning SourceEnforcement
Compiler warnings--warnings-as-errors
Dialyzer warningsZero tolerance
Credo issues--strict mode
Runtime warningsLogged and fixed

Compilation Standard:

# Required for every commit
mix compile --warnings-as-errors --force

# Required for every PR
mix dialyzer --format dialyxir

# Required for code style
mix credo --strict
