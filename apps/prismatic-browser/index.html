Prismatic Browser

Headless browser automation for web scraping, screenshots, and JavaScript rendering

170 files Collection

Abstract

Prismatic Browser provides headless browser automation capabilities for web content collection that requires JavaScript rendering, managing a supervised pool of Chromium instances for parallel page loading, screenshot capture, DOM extraction, and interaction automation. The system is essential for OSINT collection from modern JavaScript-heavy web applications where HTTP-only crawling returns empty or incomplete content. The architecture implements a pool manager that maintains configurable browser instances with automatic recycling, health monitoring, and memory-aware scaling. Anti-detection features include user-agent rotation, fingerprint management, proxy rotation with health checking, and request timing randomization to avoid triggering anti-bot protections on target sites.

1. Introduction

1.1 Problem Statement

Modern web applications render content dynamically through JavaScript execution. A standard HTTP client retrieving the HTML source of such applications receives only the application shell without rendered content. Government portals, financial services, and corporate websites increasingly use single-page application architectures that load data asynchronously after initial page load. OSINT collection from these sources requires a full browser environment that executes JavaScript, waits for content rendering, and then extracts the final DOM state.

Prismatic Browser provides this capability as a managed pool of headless browsers, abstracting the complexity of browser lifecycle management, resource control, and anti-detection measures.

1.2 Design Goals

  1. Managed browser pool -- configurable pool of headless Chromium instances with automatic lifecycle management.
  2. JavaScript rendering -- full browser execution environment for dynamic content rendering.
  3. Content extraction -- CSS selector and XPath-based data extraction from rendered DOM.
  4. Screenshot capture -- full-page and element-level screenshot capability at configurable resolutions.
  5. Anti-detection -- user-agent rotation, fingerprint management, and proxy support to avoid bot detection.
  6. Resource control -- memory limits, instance recycling, and request queuing to prevent resource exhaustion.

1.3 Scope

Prismatic Browser covers browser-based content collection. Standard HTTP crawling is handled by Prismatic Crawler Core. Browser automation for form submission and multi-step interaction is supported but not the primary use case.

2.4 Data Flow

Requests enter the RequestQueue with priority ordering. The PoolManager allocates an available browser instance (or queues the request if all instances are busy). The instance navigates to the target URL, executes JavaScript, waits for the specified selector or timeout, and performs the requested operation (extraction, screenshot, or PDF generation). Results are returned to the caller, and the instance is recycled (cleared of cookies, cache, and state) for reuse.

3. Implementation

3.1 Key Algorithms

Instance Recycling. After each request, browser instances are recycled by clearing cookies, local storage, and navigation history. Instances are fully restarted after a configurable number of requests (default 100) to prevent memory leaks from accumulating JavaScript heap usage.

Anti-Detection. Each request uses a randomly selected user-agent string from a pool of current browser versions. Browser fingerprint attributes (canvas, WebGL, fonts) are randomized. Request timing includes configurable random delays to simulate human browsing patterns.

3.3 API Surface

# Render and extract content
@spec render(String.t(), keyword()) :: {:ok, RenderResult.t()} | {:error, term()}
PrismaticBrowser.render("https://example.com",
  wait_for: "#main-content", extract: :text)

# Capture screenshot
@spec screenshot(String.t(), keyword()) :: {:ok, binary()} | {:error, term()}
PrismaticBrowser.screenshot("https://example.com",
  width: 1920, height: 1080, format: :png)

# Extract structured data
@spec extract(String.t(), map()) :: {:ok, map()} | {:error, term()}
PrismaticBrowser.extract("https://example.com", %{
  title: "h1", items: "ul.results li", links: "a[href]"})

6. Testing Strategy

6.1 Unit Tests

Extractor tests verify CSS selector and XPath extraction against known HTML structures. Anti-detection tests verify user-agent rotation and fingerprint randomization.

6.2 Integration Tests

Full rendering pipeline tests load known web pages and verify content extraction, screenshot capture, and error handling.

6.3 Property-Based Testing

StreamData generators produce random CSS selectors and URLs to verify that the browser pool handles all inputs without crashing or leaking instances.

7. Security Considerations

7.1 Threat Model

Malicious web pages could attempt to exploit browser vulnerabilities. Sandboxed Chromium execution with restricted permissions mitigates this risk. No file system access or network access beyond the target page is permitted from the browser context.

7.2 Access Control

Browser automation requires osint_query permission. All rendered content is subject to the platform's data handling policies.

8. Operational Considerations

8.1 Deployment

Requires Chromium binary installed on the host system. The pool starts automatically on application boot.

8.2 Monitoring

Telemetry events: [:prismatic, :browser, :render], [:prismatic, :browser, :instance_recycled], [:prismatic, :browser, :pool_exhausted]. Key metrics include render latency, pool utilization, and memory consumption.

8.3 Troubleshooting

SymptomLikely CauseResolution

Memory growthInstance not recyclingCheck recycling configuration

Empty contentJavaScript rendering incompleteIncrease wait timeout or add selector wait

9. Future Work

Planned enhancements include multi-browser support (Firefox, WebKit), distributed browser pool across cluster nodes, intelligent wait strategies based on page load patterns, and integration with Prismatic HAWKEYE for screenshot-based visual analysis.
