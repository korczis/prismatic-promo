Prismatic Core

Foundation module providing core abstractions, protocols, and shared utilities

180 files Foundation

Abstract

Prismatic Core is the foundational module of the Prismatic Platform, defining core abstractions, shared protocols, base behaviors, and utility functions that every other application depends on. The module establishes the platform's type system through entity protocols for consistent data access, serialization protocols for storage adapter compatibility, validation protocols for data integrity, and comparison protocols for ordering. It provides platform-wide configuration management, structured error types following the {:ok, result} | {:error, reason} convention, cryptographic utilities (hashing, HMAC, encryption), string manipulation with Czech diacritics support, date/time handling with timezone awareness, and collection utilities. Prismatic Core has zero external dependencies beyond the Elixir standard library and OTP, ensuring it can be depended upon by every application in the umbrella without introducing transitive dependency conflicts.

1. Introduction

1.1 Problem Statement

A platform comprising 90 OTP applications requires a shared foundation of types, protocols, and utilities. Without centralization, each application would define its own entity representations, error types, and utility functions, leading to incompatible interfaces, duplicated code, and inconsistent behavior. Cross-application communication requires agreed-upon data structures and protocols.

Prismatic Core provides this shared foundation, ensuring all applications speak the same language for entities, errors, configuration, and common operations.

1.2 Design Goals

  1. Zero external dependencies -- Core depends only on Elixir and OTP, preventing transitive dependency conflicts.
  2. Protocol-driven design -- shared behaviors defined as Elixir protocols, enabling polymorphic dispatch.
  3. Consistent error handling -- structured error types with {:ok, _} | {:error, _} convention enforced platform-wide.
  4. Czech language support -- string utilities handle diacritics, ICO/DIC formats, and Czech date conventions.
  5. Minimal surface area -- only genuinely shared abstractions belong in Core; domain-specific logic stays in domain applications.

1.3 Scope

Prismatic Core covers foundational abstractions shared across all applications. Domain-specific entities, business logic, and application-specific utilities reside in their respective applications.

2.3 Process Topology

Prismatic Core is a pure library application with no processes. All functions are stateless and thread-safe. No supervision tree, no GenServers, no ETS tables.

2.4 Data Flow

Applications import Core modules and call functions directly. Protocol dispatch is resolved at compile time for known types and at runtime for dynamic dispatch.

3. Implementation

3.1 Key Algorithms

Czech Diacritics Normalization. The string utility provides bidirectional conversion between accented Czech characters and their ASCII equivalents (e.g., "Praze" to "Praze", "Cesky Krumlov" to "Cesky Krumlov"), enabling fuzzy matching in entity resolution.

Structured Error Types. All errors are represented as structs with a type atom, a human-readable message, and optional details map. This enables pattern matching on error types in consuming applications.

3.2 Data Structures

defprotocol PrismaticCore.Entity do
  @spec id(t()) :: String.t()
  def id(entity)

  @spec type(t()) :: atom()
  def type(entity)

  @spec metadata(t()) :: map()
  def metadata(entity)
end

defmodule PrismaticCore.Error do
  @type t :: %__MODULE__{
    type: atom(),
    message: String.t(),
    details: map()
  }
  defstruct [:type, :message, details: %{}]

  def not_found(resource, id), do: %__MODULE__{type: :not_found, message: "#{resource} #{id} not found"}
  def validation_failed(errors), do: %__MODULE__{type: :validation_failed, details: %{errors: errors}}
  def unauthorized(reason), do: %__MODULE__{type: :unauthorized, message: reason}
end

3.3 API Surface

# Entity protocol
defimpl PrismaticCore.Entity, for: MyStruct do
  def id(entity), do: entity.uuid
  def type(_entity), do: :my_entity
  def metadata(entity), do: %{created_at: entity.inserted_at}
end

# Configuration
config = PrismaticCore.Config.get(:prismatic, :feature_flags)
PrismaticCore.Config.get(:prismatic, :max_retries, 3)

# Error handling
{:error, PrismaticCore.Error.not_found("User", user_id)}

# Crypto
hash = PrismaticCore.Crypto.sha256(data)
hmac = PrismaticCore.Crypto.hmac_sha256(key, message)

# String utilities
normalized = PrismaticCore.Utils.String.normalize_czech("Zakon o kyberneticke bezpecnosti")

6. Testing Strategy

6.1 Unit Tests

Protocol implementation tests verify correct dispatch for all supported types. Error type tests verify pattern matching and message formatting. Crypto tests verify hash computation against known test vectors.

6.2 Integration Tests

Cross-application integration tests verify that entities serialized by one application can be deserialized by another through the shared protocols.

6.3 Property-Based Testing

StreamData generators produce random strings, dates, and entity data to verify that serialization roundtrips preserve data integrity and that Czech normalization is idempotent.

7. Security Considerations

7.1 Threat Model

Crypto utilities must use constant-time comparison to prevent timing attacks. Hash functions must use cryptographically secure algorithms.

7.2 Access Control

As a library, Core has no access control. Security is enforced by consuming applications.

8. Operational Considerations

8.1 Deployment

Deploys as part of the umbrella release. No external dependencies or configuration required.

8.2 Monitoring

No telemetry (pure library). Monitoring is performed by consuming applications.
