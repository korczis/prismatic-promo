Orchestration & Multi-Agent Flow

Abstract

The Prismatic Platform's orchestration engine coordinates 434 autonomous agents across 14 specialized domains, enabling complex multi-agent workflows that transform impossible objectives into routine operations. Built on Elixir/OTP's battle-tested concurrency primitives, the orchestration layer leverages GenServer-based session lifecycle management, a five-level agent hierarchy (L1 Specialist through L5 Supreme), and a biologically-inspired mycelial network for bidirectional knowledge propagation. This document provides a comprehensive technical exploration of the orchestration architecture, including the AIAD specification framework, swarm intelligence patterns, Mind-Meld synthesis protocols, stack-based conversation tracking, supervision tree design, and the /orchestrate command that delivers 10x operational efficiency.

Introduction

The Challenge of Multi-Agent Coordination

Traditional software architectures decompose complexity through layered abstractions: controllers delegate to services, services call repositories, repositories access databases. This sequential pipeline model works for simple request-response cycles but collapses when tasks require parallel reasoning across multiple knowledge domains, real-time adaptation to changing conditions, and synthesis of contradictory evidence streams.

The Prismatic Platform faces a fundamentally different challenge. With 434 agents spanning domains as diverse as OSINT intelligence gathering, formal mathematical verification, security threat modeling, and autonomous code evolution, the orchestration problem is not merely routing requests to handlers. It is coordinating emergent intelligence: ensuring that specialist agents contribute their domain expertise, that contradictions are preserved rather than suppressed (per the NABLA axioms), and that quality gates enforce production-readiness at every stage (per the No Mercy doctrine).

Design Principles

The orchestration engine is governed by four non-negotiable design principles:

  1. OTP-Native Architecture: Every stateful entity runs in its own process under explicit supervision. No shared mutable state. No thread pools. No locks. The BEAM virtual machine's preemptive scheduling ensures fair resource allocation across thousands of concurrent agent processes.

  2. Evidence-Based Decision Making: Orchestration decisions are grounded in the Trinity Gate framework. No agent is dispatched, no swarm is formed, no result is synthesized without structural, logical, and formal verification passing all three gates.

  3. Biological Resilience: The mycelial network pattern, inspired by fungal communication networks in forests, enables bidirectional pattern propagation between agents. Knowledge flows both top-down (from coordinators to specialists) and bottom-up (from field agents to strategic commanders).

  4. Zero-Compromise Quality: The quality gates are integrated directly into the orchestration pipeline. A workflow that would produce output violating quality standards is halted before execution, not cleaned up after the fact.

The Life of an Orchestrated Request

Phase 1: Ingestion and Analysis

Every request entering the orchestration layer passes through the Ingestion Gate, a multi-stage analysis pipeline that normalizes input, assesses intent, and enriches context before any agent is activated.

Safety Validation: The request is validated against the AI Safety constraints module. Requests that violate safety boundaries are rejected immediately with structured error responses. No downstream agent ever receives an unsafe request.

Intent Recognition: Natural Language Understanding models determine the core objective. The system classifies requests across a taxonomy of 210+ command types, matching user intent to the most appropriate command specification.

Context Fusion: The request is enriched with session history from the session lifecycle manager, user context from the stack-based conversation system, and relevant knowledge from the mycelial network. This fused context ensures agents have complete situational awareness.

Complexity Assessment: A complexity scoring algorithm evaluates the request across dimensions including domain breadth (how many knowledge domains are involved), depth (how deep the reasoning must go), uncertainty (how much unknown territory exists), and time sensitivity. This score determines whether the request receives direct routing or triggers a full swarm formation.

Once the Ingestion Gate completes its analysis, the request enters the Orchestration Matrix. The Unified Orchestrator evaluates three routing strategies:

Direct Routing (complexity score below 3): Simple, single-domain requests are routed directly to the most capable specialist agent. Status checks, information retrieval, and well-defined transformations follow this path. Latency is minimal because no coordination overhead is incurred.

Team Formation (complexity score 3 to 7): Medium-complexity tasks assemble a focused team of 2-5 agents from related domains. The orchestrator selects agents based on capability matching (which agents have the required skills), availability (current load and queue depth), and historical performance (which agents have performed best on similar tasks). Team members communicate through a shared context board managed by PubSub.

Swarm Formation (complexity score 7 and above): The most complex requests trigger full swarm intelligence. The Swarm Evolution Coordinator assembles 5 or more agents from multiple domains, establishes communication channels, designates a lead agent, and activates the Mind-Meld synthesis protocol.

Phase 3: Agent Collaboration -- The Mind-Meld Protocol

The Mind-Meld is the Prismatic Platform's proprietary protocol for deep agent collaboration. Unlike simple fan-out/fan-in patterns where agents work independently and results are merged, the Mind-Meld enables continuous knowledge sharing during execution.

Step 1 -- Proposal: The designated Lead Agent analyzes the fused context and proposes a decomposition plan. This plan specifies which sub-tasks each agent will handle, what information each agent needs from others, and the expected output format.

Step 2 -- Critique: A dedicated Critic Agent reviews the proposal for logical gaps, missing edge cases, potential quality violations, and inefficient resource allocation. The Critic operates under No Doubts doctrine: full investigation before action.

Step 3 -- Refinement: The plan iterates through a rapid refinement cycle. Agents contribute domain-specific insights that reshape the approach. This phase typically completes in milliseconds thanks to BEAM's lightweight process messaging.

Step 4 -- Parallel Execution: Agents execute their assigned sub-tasks concurrently. The BEAM's preemptive scheduler ensures fair CPU allocation across all agent processes. Each agent reports progress through telemetry events, enabling real-time monitoring.

Step 5 -- Continuous Synchronization: Unlike batch-mode fan-out patterns, Mind-Meld agents continuously share intermediate results through PubSub channels. An agent discovering relevant information broadcasts it immediately, allowing other agents to adjust their approach in real-time.

defmodule PrismaticAgents.MindMeld do
  @moduledoc """
  Mind-Meld synthesis protocol for deep multi-agent collaboration.
  Agents share a continuous context stream rather than working in isolation.
  """

  alias PrismaticAgents.{SwarmRegistry, ContextBoard, QualityGate}

  @type swarm_id :: String.t()
  @type agent_result :: {:ok, map()} | {:error, term()}

  @spec initiate(swarm_id(), list(module()), map()) ::
          {:ok, map()} | {:error, term()}
  def initiate(swarm_id, agent_modules, context) do
    with {:ok, board} <- ContextBoard.create(swarm_id, context),
         {:ok, agents} <- spawn_swarm_agents(agent_modules, board),
         {:ok, proposal} <- request_proposal(agents.lead, board),
         {:ok, critique} <- request_critique(agents.critic, proposal),
         {:ok, refined_plan} <- refine_plan(agents, proposal, critique),
         {:ok, results} <- execute_parallel(agents, refined_plan, board),
         {:ok, synthesis} <- synthesize_results(results, board),
         :ok <- QualityGate.validate(synthesis) do
      {:ok, synthesis}
    end
  end

  defp spawn_swarm_agents(modules, board) do
    agents =
      modules
      |> Enum.map(fn mod ->
        {:ok, pid} = DynamicSupervisor.start_child(
          PrismaticAgents.SwarmSupervisor,
          {mod, context_board: board}
        )
        {mod.role(), pid}
      end)
      |> Map.new()

    {:ok, agents}
  end

  defp execute_parallel(agents, plan, board) do
    agents
    |> Map.values()
    |> Task.async_stream(
      fn pid -> GenServer.call(pid, {:execute, plan, board}, :infinity) end,
      max_concurrency: System.schedulers_online(),
      ordered: false
    )
    |> Enum.reduce({:ok, []}, fn
      {:ok, {:ok, result}}, {:ok, acc} -> {:ok, [result | acc]}
      {:ok, {:error, reason}}, _ -> {:error, reason}
      {:exit, reason}, _ -> {:error, {:agent_crash, reason}}
    end)
  end
end

Phase 4: Synthesis and Response

The Synthesis Engine gathers results from all participating agents and produces a unified response. This phase involves three critical operations:

Conflict Resolution: When multiple agents produce contradictory results, the engine does not simply pick the majority opinion. Following the NABLA axioms and specifically the Contradiction Preservation axiom, both perspectives are preserved with confidence scores and evidence chains. The consumer of the result receives the full picture, not a sanitized consensus.

Quality Validation: The synthesized result passes through the platform's quality gates before delivery. These gates verify structural correctness, logical consistency, and formal necessity (the three pillars of the Trinity Gate). If validation fails, the result enters a refinement loop rather than being delivered in a degraded state.

Output Formatting: The final validated result is formatted according to the requester's preferred output format (structured data, natural language, code, JSON) and augmented with provenance metadata tracing which agents contributed to which parts of the response.

Propagation Mechanics

The Mycelial Network Coordinator manages three types of signal propagation:

Intra-Domain Signals: Within a single domain (e.g., Security), agents share discoveries immediately through dedicated PubSub topics. When the Red Epistemic Attacker identifies a novel attack vector, that signal propagates to the Blue Drift Detector and Purple Closure Analyst within milliseconds.

Cross-Domain Signals: When a pattern discovered in one domain has relevance to another, cross-domain propagation bridges the gap. For example, a quality pattern identified by the QDP Enforcer that reveals a security vulnerability is propagated to the Security domain's Blue Commander for defensive assessment.

Emergence Signals: The Mycelial Emergence Sentinel monitors the network for emergent patterns -- signals that are individually insignificant but collectively indicate a systemic trend. These emergence signals trigger L3 Strategic Commander attention for campaign-level response.

Network Health and Repair

The Mycelial Healer Specialist continuously monitors network health metrics: signal latency, propagation success rates, topology efficiency, and node availability. When the healer detects degradation (a node becoming unresponsive, a signal path experiencing high latency), it initiates automated repair:

defmodule PrismaticMycelial.Healer do
  use GenServer

  @health_check_interval :timer.seconds(30)

  def handle_info(:health_check, state) do
    state.topology
    |> Enum.map(fn {node_id, node} ->
      {node_id, check_node_health(node)}
    end)
    |> Enum.filter(fn {_id, health} -> health.status == :degraded end)
    |> Enum.each(fn {node_id, health} ->
      :telemetry.execute(
        [:prismatic, :mycelial, :repair],
        %{latency_ms: health.latency_ms},
        %{node_id: node_id, reason: health.degradation_reason}
      )
      repair_node(node_id, health)
    end)

    Process.send_after(self(), :health_check, @health_check_interval)
    {:noreply, state}
  end
end

The network maintains a 99.8% signal success rate across all propagation types, with automatic rerouting around degraded nodes ensuring zero signal loss during repair operations.

Swarm Intelligence Patterns

Formation Strategies

Swarm intelligence in the Prismatic Platform goes beyond simple task distribution. It implements genuinely emergent behavior where the collective intelligence of the swarm exceeds the sum of individual agent capabilities. Three primary formation strategies are employed:

Convergent Swarm: All agents work toward a single synthesis point. Used when the objective is well-defined but requires multi-domain expertise. Example: comprehensive security assessment of a new feature, requiring code analysis, dependency audit, compliance check, and threat modeling agents to converge on a unified risk score.

Divergent Swarm: Agents deliberately explore different solution paths in parallel. Used when the problem space is poorly understood and multiple approaches should be evaluated simultaneously. Example: investigating an unknown data pattern where OSINT, statistical analysis, and graph analysis agents each pursue independent hypotheses.

Adaptive Swarm: The swarm dynamically restructures itself as new information emerges. Agents can recruit additional specialists, dismiss irrelevant participants, and shift the lead role based on which domain proves most relevant. This is the most sophisticated formation and is triggered for complexity scores above 9.

Resource Allocation

The orchestrator manages finite resources (CPU, memory, concurrent process limits) across competing swarms and individual agent requests. Priority-based allocation follows these rules:

Priority LevelAllocation StrategyPre-emption
P0 Critical (L5 Supreme missions)Unlimited resources, pre-empt lower prioritiesYes
P1 Production (security, quality gates)Reserved capacity pool, guaranteed minimumsLimited
P2 Standard (feature work, analysis)Fair-share scheduling, elastic scalingNo

The BEAM's scheduler handles CPU fairness natively through reduction-based preemption, but memory and process count limits are enforced by the orchestration layer to prevent runaway agents from destabilizing the platform.

Session Lifecycle Management

GenServer-Based Architecture

Session lifecycle management is implemented as a GenServer within the PrismaticClaude application. The SessionLifecycle module provides OTP-compliant session tracking with hook-based extensibility, circuit breaker protection, and telemetry integration.

defmodule PrismaticClaude.SessionLifecycle do
  use GenServer

  @moduledoc """
  OTP-compliant session lifecycle manager.
  Triggers mandatory evolution hooks at session boundaries.
  Implements circuit breaker pattern for hook resilience.
  """

  defstruct [
    :session_id,
    :started_at,
    :hooks,
    :circuit_breaker,
    :frame_stack,
    phase: :initialized
  ]

  # Mandatory lifecycle phases
  @phases [:initialized, :starting, :active, :pre_command,
           :post_command, :ending, :completed]

  def start_link(opts) do
    GenServer.start_link(__MODULE__, opts, name: __MODULE__)
  end

  @impl true
  def init(opts) do
    state = %__MODULE__{
      session_id: generate_session_id(),
      started_at: DateTime.utc_now(),
      hooks: load_hooks(opts),
      circuit_breaker: CircuitBreaker.new(threshold: 3, reset_after: 60_000),
      frame_stack: []
    }

    :telemetry.execute(
      [:prismatic_claude, :session_lifecycle, :init],
      %{timestamp: System.monotonic_time()},
      %{session_id: state.session_id}
    )

    {:ok, state}
  end

  def trigger(phase) when phase in @phases do
    GenServer.call(__MODULE__, {:trigger, phase})
  end
end

Lifecycle Phases

Every session follows a mandatory lifecycle with hook execution at each transition:

  1. Session Start: Executes mix autoheal.baseline and mix autoevolve status. Loads context from previous sessions. The circuit breaker pattern prevents cascading failures if mix tasks are unavailable.

  2. Pre-Command: Before any command execution, mix quality.gates.check --fast runs as a blocking gate. This ensures the platform is in a healthy state before any work begins.

  3. Post-Command: After command completion, mix autoevolve.scan --quick runs automatically. Every interaction is an opportunity for platform improvement.

  4. Session End: Executes mix autoheal.cycle and mix autoevolve.mega for comprehensive evolution. Session context is persisted to disk for cross-session continuity.

The SessionHooks module provides default implementations for all lifecycle hooks, with priority-based ordering (0-100) and enable/disable control per hook. Custom hooks can be registered at runtime for application-specific lifecycle behavior.

Stack-Based Conversation Mode

Design Rationale

Traditional conversation models treat dialogue as a flat sequence of messages. This breaks down for complex technical sessions where the user needs to explore branches, backtrack from dead ends, and maintain precise context boundaries. The stack-based conversation mode provides a structured alternative, modeling conversation as an immutable stack of frames with explicit branch and checkpoint semantics.

Frame Structure

Every response in a conversation creates an immutable frame on the stack:

FieldDescription
Frame IDMonotonically increasing identifier
User Input Summary1-2 line summary of the user's request
Assistant Output Summary1-3 line summary of the response
Key AssumptionsAssumptions made during this frame
Key DecisionsDecisions taken that affect subsequent frames
TimestampUTC timestamp of frame creation

Frame immutability is a hard invariant: once created, a frame's content cannot be modified. This ensures that conversation history is a reliable audit trail.

Stack Operations

The Stack Mode Coordinator manages six stack control commands:

CommandFunctionEffect
/stackDisplay all framesRead-only view of the complete conversation stack
/frame NShow frame N detailsInspect a specific frame's assumptions and decisions
/pop NRemove last N framesDestructive: removes context, narrows the active state
/fork NBranch from frame NDestructive: creates a new branch, preserving the parent
/checkpoint nameMark current frameCreates a named restore point for future navigation
/goto nameRestore to checkpointReturns the conversation to a previously checkpointed state

The GenServer implementation (StackConversation) uses ETS for frame storage with disk persistence to .claude/stack-conversation/, ensuring stack state survives process restarts.

Behavioral Rules

Stack-based conversation enforces strict behavioral constraints:

  • Stack-Only Context: The active conversation state is defined exclusively by the current stack. Frames below a pop boundary are treated as if they never existed.
  • No Cross-Branch Merging: Knowledge from a forked branch does not leak into other branches. Each branch maintains complete isolation.
  • Explicit Dependencies: If a request depends on information from a popped frame, the system explicitly states: "That information is no longer in the active stack."

Supervision Tree Architecture

OTP Supervision Design

The orchestration layer's supervision tree follows Elixir/OTP best practices with a hierarchical structure that isolates failure domains and enables targeted recovery.

defmodule PrismaticAgents.Application do
  use Application

  @impl true
  def start(_type, _args) do
    children = [
      # Core registries (must start first)
      {Registry, keys: :unique, name: PrismaticAgents.AgentRegistry},
      {Registry, keys: :duplicate, name: PrismaticAgents.PubSub},

      # Static agent pool (long-lived agents)
      {PrismaticAgents.StaticSupervisor, strategy: :one_for_one},

      # Dynamic swarm supervisor (short-lived swarm agents)
      {DynamicSupervisor,
        strategy: :one_for_one,
        name: PrismaticAgents.SwarmSupervisor,
        max_children: 1000},

      # Orchestration engine
      PrismaticAgents.Orchestrator,

      # Mycelial network
      PrismaticAgents.MycelialCoordinator,

      # Session management
      PrismaticClaude.SessionLifecycle,
      PrismaticClaude.StackConversation,

      # Telemetry reporters
      PrismaticAgents.TelemetryReporter
    ]

    opts = [
      strategy: :rest_for_one,
      name: PrismaticAgents.Supervisor,
      max_restarts: 10,
      max_seconds: 60
    ]

    Supervisor.start_link(children, opts)
  end
end

The rest_for_one strategy ensures that if a foundational component (like the Agent Registry) crashes and restarts, all components that depend on it are also restarted in the correct order. The max_children: 1000 limit on the SwarmSupervisor prevents runaway swarm formation from exhausting system resources.

Failure Isolation

Each domain's agents run under their own sub-supervisor, ensuring that a crash in the Security domain does not affect Intelligence or Evolution agents. The BEAM's "let it crash" philosophy is fully embraced: rather than defensive coding with try/catch blocks, agents are designed to crash cleanly and restart with fresh state.

The /orchestrate Command

10x Efficiency Through Intelligent Coordination

The /orchestrate command is the primary interface for invoking the full orchestration engine. It provides 10x efficiency over manual multi-agent coordination by automating swarm formation, context fusion, quality gate enforcement, and result synthesis.

# Basic orchestration
/orchestrate "Analyze the security posture of the authentication module"

# Orchestration with domain constraints
/orchestrate --domains=security,code-quality "Review PR #1234 for vulnerabilities"

# Orchestration with priority override
/orchestrate --priority=P0 "Critical production incident in perimeter service"

# RC1 orchestration (release candidate quality)
/rc1-orchestrate "Prepare prismatic_perimeter for production deployment"

When invoked, the /orchestrate command:

  1. Analyzes the request through the Ingestion Gate
  2. Selects the optimal formation strategy (direct, team, or swarm)
  3. Assembles agents based on capability matching and domain relevance
  4. Initiates the Mind-Meld protocol for complex requests
  5. Enforces all six quality gates during execution
  6. Synthesizes results with full provenance tracking
  7. Delivers the response with confidence scores and evidence chains

The related /rc1-orchestrate command applies stricter quality thresholds appropriate for release candidate workflows, where zero defects are mandatory.

Cross-Domain Orchestration Patterns

Pattern: Intelligence-to-Security Pipeline

When an OSINT investigation discovers a potential threat indicator, the orchestration engine automatically routes the finding through the security color-team pipeline:

  1. Intelligence domain agents discover the indicator
  2. Mycelial cross-domain signal propagates to Security domain
  3. Red team receives the indicator for adversarial simulation
  4. Blue team assesses defensive posture against the simulated attack
  5. Purple team synthesizes findings and evaluates closure conditions
  6. White team formally verifies that mitigations hold

This entire pipeline executes without human intervention, driven by the orchestration engine's cross-domain routing rules and the mycelial network's signal propagation mechanics.

Pattern: Quality-Driven Evolution

When quality gates detect a degradation trend (even if individual gates still pass), the orchestration engine triggers autonomous self-healing:

  1. Quality domain signals detect trend
  2. Evolution domain receives cross-domain propagation
  3. Darwinian Evolution Coordinator generates candidate fixes
  4. Quality gates evaluate each candidate
  5. The fittest candidate is selected and applied
  6. Regression tests verify the fix does not break existing functionality
